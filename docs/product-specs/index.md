# Product Specs Index

Product specs define user-facing behavior and acceptance criteria.

## Core user flow

1. User pairs Ghost Operator via Bluetooth (or plugs in USB)
2. Device sends periodic keystrokes and mouse movements to prevent idle/lock
3. User optionally configures via encoder menu or web dashboard
4. Device runs unattended for hours/days (scheduled sleep/wake optional)

## Writing specs

Use the [template](_template.md). Each spec should define:
- **User story:** Who wants what and why
- **Acceptance criteria:** Observable behaviors that must be true
- **Edge cases:** What happens when input is invalid, services fail, or timeouts occur
- **Not in scope:** Explicitly state what this spec does NOT cover
